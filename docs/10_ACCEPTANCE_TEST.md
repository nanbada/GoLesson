# 10. ACCEPTANCE TEST — 출시 승인 기준

자동 E2E 없음(파일럿) — 본 문서의 수동 시나리오가 QA 기준. 각 항목은 REQ/BR 판정 근거를 참조한다.
발송 테스트는 반드시 테스트 학생(GoAlimi 출결번호 **7707 신성화** = 운영자 본인 카톡)으로만.

## 0. 테스트 데이터

모든 시나리오는 `aidd_docs/fixtures/mvp-seed-data.md`의 고정 seed(학생 3·교재 5·별칭·스케줄·직전 진도·과제·결제)를 로드한 상태를 전제로 한다. §3 고정 문장과 seed는 짝으로 관리 — 한쪽을 바꾸면 다른 쪽도 갱신.

## 1. 시나리오 테스트

### T1. 수업 기본 흐름 (REQ-202~205, 303~305, 401~404)

| # | 절차 | 통과 기준 |
|---|---|---|
| 1 | 폰에서 PWA 열기 | 3초 내 오늘 화면, 오늘 요일 수업이 시간순 표시 |
| 2 | 학생 카드 탭 → [수업 시작] | 직전 수업 요약 표시, 시작시각 기록 |
| 3 | 진도 to에 직전보다 큰 값 입력 → 저장 | 2초 내 저장, from은 자동이었음 |
| 4 | 과제 1건 추가(학원내) → 완료 체크+코멘트 | 저장됨 |
| 5 | [수업 완료] | 종료시각·수업시간 기록, 오늘 화면에 ✓ |
| 6 | 전체 소요 측정 | 기록 조작 합계 30초 이내 (REQ-902) |

### T2. 진도 경계 (BR-102~106)

| # | 절차 | 통과 기준 |
|---|---|---|
| 1 | to에 직전 진도보다 낮은 값 | 경고 표시되나 저장은 가능(복습) |
| 2 | 직전과 동일 구간 재저장 | "동일 진도" 안내 + 저장 가능, 로그 2건 존재 |
| 3 | to < from 입력 | 저장 거부 + 안내 |
| 4 | to ≥ 총량 입력 | 완료 확인 다이얼로그 → 승인 시 completed |
| 5 | 완료 교재에 진도 입력 시도 | 불가, 다음 교재 배정 유도 |

### T3. 과제 이월 (BR-303~305)

집 과제(take_home) 등록 → 다음 수업 화면에 자동 표시 → 미완료 체크 → 재부여 시 새 행 생성, 이력 2건 모두 조회 가능.

### T4. 빠른 입력 (REQ-501~508)

| # | 절차 | 통과 기준 |
|---|---|---|
| 1 | 정형 3줄 붙여넣기 → 파싱 | 3장 카드, 전부 method=regex, AI 미호출 |
| 2 | 비정형 1줄 포함 | 해당 줄만 AI, 나머지 regex |
| 3 | 없는 학생명 포함 | 그 줄만 오류 카드, 드롭다운 수동 지정 가능 |
| 4 | [일괄 저장] | 각 줄이 당일 lesson+진도+과제+코멘트로 저장 |
| 5 | 고정 테스트 문장 20개 (§3) | 5요소 정확 매핑 19/20 이상 (95%, REQ-508) |
| 6 | 저장 후 parse_logs 확인 | 저장된 줄의 status=confirmed로 갱신됨 (04 §5 5-1 예외 경로) |

### T5. 리포트 (REQ-601~607, BR-401~406)

| # | 절차 | 통과 기준 |
|---|---|---|
| 1 | 2주 기간, 학생 3명 생성 | 10초/명 내 draft 3건, 수치가 실데이터와 일치(수기 대조) |
| 2 | 본문에 AI 의견 확인 | 입력 코멘트에 없는 사실·수치 없음 |
| 3 | 코멘트 없는 과목 | 해당 의견 섹션 생략 |
| 4 | 본문 수정 → [발송 승인] | ready 전환, 수정 반영 |
| 5 | 같은 학생·기간 재생성 | 새 draft 중복 생성 안 됨(기존 갱신) |

### T6. 발송 파이프라인 (REQ-701~707, BR-501~505) — 테스트 계정만

| # | 절차 | 통과 기준 |
|---|---|---|
| 1 | ready 리포트 [발송] | outbox pending 생성, dedupe_key 확인 |
| 2 | [발송] 연타 | 중복 등록 안 됨(409) |
| 3 | Bridge 폴링 대기 | **2분 내** processing→sent (poll 60초 + GoAlimi 발송 최대 40초 감안), 본인 카톡에 리포트 수신, 600~900자 온전 |
| 4 | reports 상태 | sent + 발송시각. 본문 UI 읽기 전용 + DB 직접 update도 거부(t_reports_immutable 트리거, REQ-1004) |
| 5 | [재발송] | v2 키로 신규 발송 성공 |
| 6 | GoAlimi 중지 후 발송 | pending 유지(유실 없음) → GoAlimi 기동 후 다음 주기 발송 |
| 7 | 대표 학부모 없는 학생 | 발송 버튼 비활성 + 사유 표시 |
| 8 | 21시 이후 발송 등록 | pending 유지, 익일 09시 이후 발송 |

### T7. 수강료 (REQ-801~805)

입력 30초 내 완료 / 월별 합계 = 수기 계산 일치 / 수정 시 audits에 전후 기록 / 음수 항목(환불) 집계 반영.

### T8. 동기화 (REQ-105·106, BR-701~703)

GoAlimi에 테스트 학생 등록 → 10분 내 GoLesson 표시 / GoAlimi에서 비활성화 → 목록 제외·기록 유지 / Bridge 중지 24시간 조건에서 경고 배너.

### T9. 모바일·네트워크 (REQ-901~906, 951~952)

320px 세로 가로스크롤 없음 / 버튼 44px+ / 홈 화면 설치·앱 모드 실행 / 비행기 모드에서 진도 저장 → 오류 표시 + 입력 보존 → 복귀 후 재제출 성공 / 작성 중 뒤로가기 → 복귀 시 draft 유지.

### T10. 권한·보안

로그아웃 상태 접근 → 로그인 화면 / anon 키로 REST 직접 조회 → 거부(RLS·GRANT) / **authenticated 계정으로 students·parents·attendance insert/update 시도 → 거부** (읽기 전용 사본, 04 §5 A군) / parse_logs: 타인이 만든 행의 status 갱신 → 거부, 본인 행의 status 외 컬럼(raw_text 등) 갱신 → 거부(컬럼 GRANT) / sent 리포트 body update → 거부(트리거) / **authenticated로 reports.status='sent' 직접 insert/update → 거부** (발송 완료는 Bridge 소유, 04 §5 3-1) / 비활성 profile 로그인 → 데이터 접근 불가 / claim_outbox RPC를 authenticated로 호출 → 거부(service_role 전용) / service_key가 git 이력·프론트 번들에 없음.

### T12. GoAlimi 확장 API·Bridge 정합 (08_GOALIMI §3~§5 — GoAlimi 측 구현 검수 겸용)

| # | 절차 | 통과 기준 |

| 7 | 발송 도중 Bridge 강제 종료 → 재기동 | stale processing이 goalimi_custom_id 조회로 sent 반영 — 중복 발송 0건 (05_API §3 회수) |
