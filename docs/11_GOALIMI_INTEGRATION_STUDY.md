# 11. GOALIMI INTEGRATION STUDY — GoAlimi 기능의 GoLesson 통합 연구

연구 질문: **"GoAlimi 전체 기능을 GoLesson 내부의 일부 서비스로 포함할 수 있는가? 포함해야 하는가?"**
결론 요약: **기술적으로 부분 가능, 그러나 파일럿 단계에서는 분리 운영이 유리. 조회성 기능은 이미 GoLesson에 흡수되어 있고, 물리적으로 이동 불가한 발송 엔진이 남는 한 완전 통합의 실익이 없다.** (§6 권고, §7 재검토 트리거)

| # | 기능 | 구현 위치 | 성격 |
|---|---|---|---|

## 2. 물리 제약 — 통합 여부와 무관하게 성립하는 사실

1. **G2(카톡 발송)는 클라우드로 이동 불가.** 카카오톡 공식 메시지 API 부재 → PC앱 UI 자동화가 유일한 무료 수단이며, 윈도우 데스크톱 세션이 필요하다. 즉 **어떤 통합 시나리오에서도 학원 PC 상주 컴포넌트는 반드시 남는다.** 유일한 대체는 알림톡 API(유료)이고, 이는 GoAlimi REFERENCE §6.6에서 "유일한 업그레이드 경로"로 이미 결론난 사항이다.
2. **G1(체크인)은 클라우드 이동 가능하나 등하원 안정성이 후퇴한다.** 현재 iPad→로컬 LAN 구조는 인터넷 장애와 무관하게 동작한다. 클라우드 PWA로 옮기면 학원 인터넷 장애 = 체크인 불가 + 등하원 알림 중단. 출결은 GoAlimi에서 가장 실패하면 안 되는 기능이다.
3. **GoAlimi는 운영 중 실서비스 + 타 학원 시범 도입 진행 중(2026-06~).** 통합을 위한 재작성은 두 학원의 운영 리스크이며, GoAlimi의 "윈도우 PC 1대 설치로 끝"이라는 독립 배포 모델(출결만 필요한 학원용)을 훼손한다.

## 3. 옵션 비교

| 옵션 | 내용 | 장점 | 단점 | 판정 |
|---|---|---|---|---|
| **A. 분리 운영 + Bridge 연동** (현 설계) | GoAlimi 독립 유지, Bridge가 발송 위임·데이터 동기화 | 리스크 최소 — 발송 자동화 코어(kakao_pc.py·큐) 무변경, 독립 제품 가치 유지. GoAlimi 측 작업은 주변 API 확장(08 §3)에 한정 | 관리 UI 2곳(학생 등록만 GoAlimi), 학원 밖 학생 등록 불가(월 1~2회 빈도), GoAlimi API 확장 선행 필요 | **채택** |
| B. 부분 흡수 확대 (학생 CRUD를 GoLesson으로) | 학생 등록·수정 UI를 GoLesson에 두고 Bridge가 GoAlimi로 역동기화 | 관리 UI 일원화, 외부에서 학생 등록 가능 | **양방향 동기화** — 충돌 규칙·장애 시 정합성 관리 필요. 체크인·발송이 로컬 DB 참조라 동기화 지연이 곧 장애. 파일럿 규모 대비 과잉 | 보류 (파일럿 후 불편이 실측되면 재검토) |
| C. 완전 통합 (GoAlimi = GoLesson 서브모듈) | 체크인 PWA·학생 마스터·출결을 전부 Supabase로, PC에는 발송 에이전트만 잔존 | 단일 시스템·단일 DB, 모든 기능 외부 접근 | 체크인 인터넷 의존(§2-2), GoAlimi 전면 재작성(실서비스 2곳 리스크), 그래도 PC 컴포넌트는 잔존(§2-1) → "완전 클라우드"는 애초에 불가능, 개발 수 주 추가, 독립 배포 모델 상실 | 기각 (파일럿 단계) |
| D. 조건부 통합 (장기 로드맵) | 알림톡 API 전환 시점에 C를 재평가 | 발송의 클라우드화로 최대 장벽 소멸 | 알림톡 비용 발생, 시점 미정 | §7 트리거로 관리 |

## 4. 기능별 처리 매핑 (옵션 A 기준 — 중복처럼 보이는 것의 정리)

| GoAlimi 기능 | GoLesson에서의 취급 | 비고 |
|---|---|---|
| G1 체크인 | 관여 안 함 (GoAlimi 전담) | |
| G2 발송 엔진 | **재사용** — Bridge → `/api/notify/custom` (08_GOALIMI) | GoLesson은 발송 로직을 새로 만들지 않는다 |
| G3 실시간 모니터링 | 흡수 안 함 — GoLesson 출결은 10분 지연 사본(조회용) | 실시간이 필요한 현장 모니터는 GoAlimi `/admin` 유지 |
| G4 학생/학부모 CRUD | 흡수 안 함 — GoAlimi 마스터, GoLesson 읽기 전용(BR-701) | 외부에서 등록 불가는 수용(월 1~2회) |
| G5 출결 조회·CSV | **사실상 흡수됨** — attendance 동기화로 GoLesson에서 기간 조회·리포트 집계 | 원장이 외부에서 출결 확인 가능 (준실시간) |
| G6 시간표 | 흡수 안 함 — 목적이 다름 | GoAlimi=학년×과목(출결 참고), GoLesson=학생별 1:1 슬롯(schedule_slots). 이름만 같은 별개 기능 |
| G7 코멘트 | **기능 중복 주의** — 수업 코멘트는 GoLesson `comments`로 일원화 | GoAlimi teacher_comments는 상담 기록용으로 남기되, 리포트에 들어갈 코멘트는 GoLesson에 입력 (운영 규칙, 온보딩 시 안내) |
| G8 휴원일 | 파일럿에서는 흡수 안 함 | 리포트 출석 집계(BR-403)는 '예정 대비 실시'라 휴원일 없어도 성립. 필요해지면 Bridge 동기화 1개 추가로 해결 가능 |
| G9 운영 도구 | Bridge가 자체 백업 추가 (09_DEPLOY §5) | GoAlimi 백업 체계는 그대로 |

## 5. 통합했을 때만 얻는 것 vs 분리로도 얻는 것

| 요구 | 분리(A)로 충족되나? |
|---|---|
| 원장이 외부에서 출결 확인 | ✅ (10분 지연 사본) |
| 리포트에 출결 자동 포함 | ✅ (동기화) |
| 카톡 발송 재사용 | ✅ (Bridge 위임) |
| 외부에서 학생 신규 등록 | ❌ — 유일한 실질 격차. 빈도(월 1~2회)와 대안(학원에서 등록, 원격데스크톱) 고려 시 수용 가능 |
| 단일 코드베이스 | ❌ — 그러나 사용자 5명 규모에서 저장소 2개는 부담이 아니라 **격리 이점**(GoAlimi 안정성 보호) |

## 6. 권고

**옵션 A(분리 운영 + Bridge 연동, 현 설계) 유지.**

근거: ① 발송 엔진이 PC를 요구하는 한 완전 통합해도 하이브리드 구조는 사라지지 않는다(통합의 기대 이익 절반이 애초에 실현 불가). ② 체크인의 오프라인 내성은 통합 시 오히려 후퇴한다. ③ GoAlimi는 독립 제품으로 타 학원 배포 중 — 흡수는 제품 전략상 손실. ④ 유일한 실질 격차(외부 학생 등록)는 빈도가 낮아 수용 가능. ⑤ 오버엔지니어링 금지 원칙(CLAUDE.md 절대규칙 1)에 따라, 실측된 불편 없이 양방향 동기화(옵션 B)를 선제 구축하지 않는다.

## 7. 재검토 트리거 (하나라도 발생 시 이 문서 §3을 다시 평가)

1. **알림톡 API 전환 결정** — 발송의 PC 종속이 사라짐 → 옵션 C의 최대 장벽 소멸. 그 시점에 남는 쟁점은 체크인 오프라인 내성뿐.
2. 파일럿 중 "학원 밖 학생 등록" 요구가 월 수회 이상으로 실측될 때 → 옵션 B 재평가.
3. GoAlimi 타 학원 배포가 항상 GoLesson 동반 설치 형태로 수렴할 때 (독립 배포 모델의 가치 소멸).
4. 학원 PC 이중 상주(GoAlimi+Bridge) 운영 장애가 반복될 때 — 이 경우 우선 Bridge를 GoAlimi 프로세스에 합치는 소통합(GoAlimi 측 워커 1개 추가)부터 검토. 단 이는 GoAlimi 프로젝트의 결정 사항.

## 8. 화면 단위 통합 검토 — "GoLesson 페이지 안에서 GoAlimi를 한 번에 보고 관리" (2026-07-03 추가)

기능 통합은 §6에서 기각·수용되었고, 남은 질문은 UI 수준: 원장/강사가 PC에서 GoLesson 웹을 쓸 때 GoAlimi 로컬 웹(`/admin`, `/student`)을 같은 화면에서 다룰 수 있는가.

### 8.1 기술 옵션 평가

| 방식 | 가능 여부 | 판정 근거 |
|---|---|---|
| **E1. iframe 임베드** (GoLesson 페이지 안에 GoAlimi 화면 삽입) | **불가** | GoLesson은 HTTPS, GoAlimi는 로컬 HTTP(`http://192.168.x.x:8000`). 브라우저의 **mixed content 정책**이 HTTPS 페이지 내 HTTP iframe을 차단한다(Chrome/Edge/Safari 공통, 사용자 설정으로만 해제 가능 — 운영 불가). 로컬 IP에는 공인 TLS 인증서 발급이 불가하고, 사설 CA(mkcert)는 모든 기기에 루트 인증서 설치가 필요해 파일럿 운영 부담 초과 |
| **E2. Cloudflare Tunnel로 GoAlimi를 HTTPS 노출 후 iframe** | 가능하나 **비권장** | 현재 GoAlimi `/admin`은 무인증(LAN 물리 보안 전제) — 인터넷 노출 시 보안 사고 직결. Cloudflare Access로 보호해도 iframe 내 인증 UX 복잡 + 자체 도메인 필요 + **인바운드 금지 원칙(08_GOALIMI §1) 위반**. 이득(임베드) 대비 비용·위험 과다 |
| **E3. 새 탭 런처** (GoLesson 메뉴에 GoAlimi 바로가기) | **가능 — 채택** | 링크 탐색은 mixed content 제한과 무관. GoLesson이 단일 진입점이 되고 GoAlimi는 한 클릭 거리. 구현 비용 ≈ 0. 제약: 학원 내 네트워크에서만 열림(외부에서는 안내 문구) |
| **E4. 조회 데이터의 GoLesson 내 표시 확대** | 가능 — 이미 대부분 반영 | 출결 조회는 동기화로 GoLesson 화면에 이미 존재(§4 G5). "보기"의 대부분은 임베드 없이 충족됨 |

참고: E3에서 "학원 내인지" 자동 감지는 불가하다 — HTTPS 페이지에서 로컬 HTTP로의 fetch(health check) 자체가 mixed content로 차단되므로, 버튼은 항상 표시하고 안내 문구로 처리한다.

### 8.2 결정

**E3(런처) + E4(조회 표시)** 조합 채택. PRD REQ-910, 03_UI_SPEC §10에 반영.

- GoLesson 더보기 메뉴에 **"GoAlimi 출결 관리"** 버튼: `app_settings.goalimi_admin_url`(기본 `http://192.168.0.100:8000/admin`)을 새 탭으로 오픈. 부가 문구 "학원 네트워크에서만 열립니다".
- 학원 PC 운영 팁(사용자 가이드에 기재): 브라우저에 GoLesson·GoAlimi 두 탭을 고정(pinned)하거나 창을 좌우 분할하면 사실상 한 화면 운영이 된다. 시스템이 강제할 사항은 아님.
- E2(터널+Access)는 "외부에서 GoAlimi 관리" 요구가 실측될 때 §7 트리거 2와 함께 재검토.

## 9. 로그인·계정 정책과의 관계 (참고)

GoLesson은 사전 등록 5계정 미만·동일 계정 다기기 동시 접속 허용(REQ-908~909)이며, GoAlimi `/admin`은 현재 LAN 접근 기반(별도 로그인 없음). 분리 운영에서는 이 차이가 자연스럽다 — GoAlimi는 학원 내 물리 보안, GoLesson은 클라우드 인증(RLS). 통합(옵션 C) 시에는 체크인 단말의 무인증 요구와 관리 화면 인증 요구를 한 앱에서 분리 설계해야 하는 부가 과제가 생긴다.
