# **1. 프로젝트 Acceptance Rule**

모든 요구사항은 다음 5가지 조건을 만족해야 합니다.

* **ID** (REQ-xxx)
* **Requirement**
* **Precondition**
* **Acceptance Criteria**
* **Priority (Must / Should / Could)**

예시

| ID           | REQ-201                 |
| :----------- | :---------------------- |
| Requirement  | 오늘 수업목록 표시              |
| Precondition | 오늘 수업이 3건 등록되어 있다.      |
| Acceptance   | 시간순으로 정렬되어 3명이 모두 표시된다. |
| Priority     | Must                    |

이 형식이면 그대로 QA 문서와 E2E 테스트 케이스의 기반으로 사용할 수 있습니다.

# **5. 추가 인터뷰 추천**

이 프로젝트는 **사용자 인터뷰의 투자 대비 효과가 매우 큰 유형**입니다.

개발 전에 아래 정도만 확인해도 UI와 기능이 크게 개선될 가능성이 높습니다.

### **인터뷰 대상**

* 원장
* 강사 1\~2명

### **질문(15\~20분)**

1. 수업 종료 후 기록에 실제 몇 분을 쓰는가?
2. 가장 자주 쓰는 메모 표현 20개는 무엇인가?
3. 가장 자주 사용하는 교재와 진도 표기 방식은? (페이지/단원/챕터)
4. 숙제는 어떤 형식으로 적는가? (페이지, 단원, 문제 번호 등)
5. 학부모가 가장 자주 묻는 질문은 무엇인가?
6. 리포트에서 반드시 들어가야 하는 항목과 거의 보지 않는 항목은 무엇인가?
7. 수강료 관리에서 가장 불편한 점은 무엇인가?
8. 휴대폰과 PC 중 어떤 기기를 가장 많이 사용하는가? 상황별로 다른가?

이 답변을 기반으로 화면을 설계하면, 불필요한 기능을 줄이고 실제 현장에서 빠르게 사용할 수 있는 UI를 만들 가능성이 크게 높아집니다.

### **현재 기준으로 추천하는 핵심 설계**

* **진도/과제:** 중복 저장 허용 + 동일 내용 저장 시 경고만 표시.
* **입력 파싱:** Regex/사전 기반 파서를 기본으로 하고, 실패한 경우에만 AI 호출(Fallback).
* **학부모 리포트:** 템플릿 기반으로 출석·진도·숙제는 코드가 생성하고, AI는 교사 코멘트 문단만 자연스럽게 다듬는 하이브리드 방식.
* **GoAlimi 연동:** 기존 `students`, `parents`, `attendance_logs`는 그대로 재사용하고 학원 관리용 테이블만 추가하는 확장형 구조.

이 구성이 구현 난이도, 운영 비용, 유지보수성 측면에서 가장 균형이 좋습니다.
